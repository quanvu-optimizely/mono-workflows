# Anti-Pattern Catalogue

Each entry has: ID, description, detection signals, bad example, and the
required corrective action. Apply ALL checks before writing any task file.

---

## ANTI-001 — Cross-Domain Task

**Description**: A single task spans multiple concern types (e.g., data model
+ API route + UI component).

**Detection signals**:
- Allowed Files contains files from ≥ 2 concern layers
- Objective describes actions across multiple architectural tiers
- Requirements mix database, business logic, and presentation concerns

**Bad example**:
```
## Allowed Files
- <schema-file> — modify
- src/app/api/<entity>/route.ts — create
- src/components/<Entity>Form.tsx — create
- src/types/<entity>.ts — create
```

**Correction**: Split into 4 tasks: schema, types, API route, component.

## ANTI-002 — God Task

**Description**: A task lists more than 5 allowed files.

**Detection signal**: Allowed Files section has > 5 entries.

**Bad example**:
```
## Allowed Files
- src/lib/<entity>.repository.ts
- src/lib/<entity>.service.ts
- src/app/api/<entity>/route.ts
- src/app/api/<entity>/[id]/route.ts
- src/components/<Entity>List.tsx
- src/components/<Entity>Card.tsx
- src/types/<entity>.ts
```
7 files — fails the gate.

**Correction**: Split by concern (schema A) or by method group (schema B).

## ANTI-003 — Vague Objective

**Description**: Objective uses non-specific, unmeasurable language.

**Detection signals** — flag ANY of these in Objective or Requirements:
- `refactor`, `improve`, `clean up`, `optimize`, `enhance`, `revamp`
- `handle`, `manage`, `deal with`, `take care of`
- `"as needed"`, `"etc."`, `"and more"`, `"other relevant"`
- `"make it work"`, `"implement properly"`, `"works correctly"`

**Bad example**:
```
## Objective
Refactor the <entity> module to improve code quality and handle edge cases
as needed. Make sure everything works correctly.
```

**Correction**: Name exact file, exported symbol, method signature:
```
## Objective
Create `src/lib/<entity>.repository.ts` exporting a `<Entity>Repository` class
with methods `findById(id: string): Promise<<Entity> | null>` and
`create(data: Create<Entity>Input): Promise<<Entity>>`. Both methods use the
DB client passed via constructor injection.
```

## ANTI-004 — Missing File Scope

**Description**: Allowed Files is empty, uses wildcards, or defers path
decisions to the executor.

**Detection signals**:
- Section is empty
- Contains: `"any relevant files"`, `"files as needed"`, `"src/**"`, `"**/*"`
- Contains `[TBD]` without a corresponding entry in Dependencies

**Bad example**:
```
## Allowed Files
- Any files related to <entity> creation
```

**Correction**: List every specific path. If a path depends on a prior task
(e.g., a generated migration), write:
```
- [TBD: path generated by TASK-002 — migration command output]
```
and list TASK-002 in Dependencies.

## ANTI-005 — Implicit Dependency

**Description**: A task imports types, functions, or modules produced by
another task, but that task is not in Dependencies.

**Detection signals**:
- Requirements say `"import <Entity> from src/types/<entity>.ts"` but no task
  creating that file appears in Dependencies
- Allowed Files contain a file that a prior task already owns (write conflict)
- Task references a named export from a file not in its Allowed Files

**Bad example**:
```
## Requirements
1. Import the <Entity> interface from src/types/<entity>.ts

## Dependencies
None
```
(TASK-002 creates `src/types/<entity>.ts` but is missing from Dependencies.)

**Correction**: Audit every import implied by Requirements. Add every producing
task to Dependencies.

## ANTI-006 — Non-Verifiable Acceptance Criteria

**Description**: Acceptance criteria use subjective language or cannot be
verified by a command.

**Detection signals**:
- `"works correctly"`, `"looks good"`, `"is implemented"`, `"functions as expected"`
- `"the user can X"` without a corresponding test or curl command
- Missing build command as the first criterion

**Bad example**:
```
## Acceptance Criteria
- [ ] The <entity> repository works correctly
- [ ] Error handling is in place
- [ ] Code is clean
```

**Correction**:
```
## Acceptance Criteria
- [ ] `<build-command>` exits with code 0
- [ ] `<type-check-command>` reports no errors in `src/lib/<entity>.repository.ts`
- [ ] Unit test `findById returns null for unknown ID` passes
```

## ANTI-007 — Architecture Drift Invitation

**Description**: Requirements give the executor autonomy to make architectural
decisions that belong to the architect.

**Detection signals**:
- `"choose an appropriate design pattern"`
- `"use best practices for error handling"`
- `"refactor as needed"`
- `"structure it however makes sense"`
- `"you may also add..."`, `"feel free to..."`

**Bad example**:
```
## Requirements
1. Implement the <entity> service using appropriate patterns
2. Handle errors as needed
3. Feel free to add caching if useful
```

**Correction**: Every architectural choice is prescribed, not deferred:
```
## Requirements
1. Export a `<Entity>Service` class (not a plain object or factory function).
2. Wrap ALL DB calls in try/catch. On error, throw
   `new AppError('DB_ERROR', err.message)` — do NOT swallow errors.
3. Do NOT add caching — that is out of scope for this task.
```

## ANTI-008 — Missing Context Update Instruction

**Description**: Context Update section is empty or absent, breaking the
story's audit trail.

**Detection signal**: Section empty, missing, or says "update context as
appropriate".

**Correction**: Every task must contain the verbatim block to append to
`context.md`. See `docs/task-format.md` § Context Update for the template.

---

## ANTI-009 — Circular Dependency

**Description**: Task A depends on Task B which depends on Task A (directly
or transitively).

**Detection signal**: A cycle is found when building the dependency graph.

**Correction**: Identify the shared artifact causing the cycle. Extract it
into a new leaf task (no dependencies) that both tasks depend on.

Example: if a Service and a Repository both import from each other (a design
flaw), create an `<Entity> Repository Interface` task (pure types, no
implementations) that both depend on, breaking the cycle.

---

## ANTI-010 — Premature Test Task

**Description**: A test task is generated before the module it tests has a
corresponding implementation task.

**Detection signal**: A test task (`TASK-NNN Tests`) has no implementation
task in its Dependencies, or the implementation task ID is higher than the
test task ID.

**Correction**: Test tasks always get the highest TASK-IDs in a story.
Add the implementation task to Dependencies.

## ANTI-UI-001 — Raw Design Value in UI Task

**Description**: A UI task (component, page) uses hard-coded colors, sizes,
spacing, or other raw design values instead of design system tokens.

**Detection signals**:
- Requirements or examples contain literal color values (hex, rgb, hsl, etc.)
- Requirements contain hard-coded pixel values for spacing/sizing
- No reference to design token names from `.ai/architecture.md § Stack Rules`

**Bad example**:
```
## Requirements
1. Set background to #1a1a2e and text color to #ffffff
```

**Correction**: Always use design system token names per your stack's conventions:
```
## Requirements
1. Apply the background and foreground token classes — never raw values.
   Token names are defined in `.ai/architecture.md § Stack Rules`.
```

---

## ANTI-UI-002 — Wrong Primitive Library

**Description**: A UI task references a UI primitive library that is not the
approved one for this project's stack.

**Detection signals**:
- Requirements reference a UI primitive import not listed in `.ai/architecture.md § Stack`
- Requirements suggest installing a UI library not approved by the project

**Correction**: Check `.ai/architecture.md § Stack Rules` for the approved UI
primitive library. Replace all references to unapproved libraries with the
correct one.

---

## ANTI-UI-003 — Missing Class Merging Utility

**Description**: A UI task does not require use of the project's approved class
merging utility, allowing the executor to use string concatenation or template literals.

**Detection signal**: Requirements for a component/page have no mention of the
class merging utility.

**Correction**: Add to Requirements:
```
N. Use the class merging utility specified in `.ai/architecture.md § Stack Rules`
   for ALL conditional class merging.
   Do NOT use template literals or string concatenation for class names.
```

## Quick Reference — Red-Flag Phrase Blocklist

Ban these from ALL task sections. If found → rewrite or split:

| Phrase | Reason banned |
|---|---|
| `refactor` | Implies scope creep |
| `improve` | Not measurable |
| `as needed` | Defers scope to executor |
| `etc.` | Implies unbounded scope |
| `any relevant files` | Violates file scope |
| `best practice` | Defers architecture to executor |
| `feel free to` | Implies executor autonomy |
| `works correctly` | Not verifiable |
| `make sure` | Not a verifiable criterion |
| `you may also` | Implies optional scope |
| `handle errors` | Underspecified — name the error type |
| `appropriate` | Defers design decision |
| `clean code` | Subjective |
| `properly` | Subjective |
