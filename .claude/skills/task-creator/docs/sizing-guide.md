# Task Sizing Guide

## The Four Gates

A task passes sizing review only if ALL four gates pass:

| Gate | Limit | Rationale |
|---|---|---|
| File count | ≤ 5 files | Bounded context for executor |
| Concern count | = 1 | Single responsibility |
| LOC estimate | ≤ 200 lines changed | Fits in executor context window |
| Upstream deps | ≤ 3 tasks | Reduces sequencing bottlenecks |

Fail any gate → split the task using the patterns below.

---

## LOC Reference Values

Use these when estimating whether a task will exceed 200 lines:

| Artifact | Typical LOC |
|---|---|
| Data model (3–6 fields) | 10–20 |
| TypeScript interface (5–10 fields) | 10–25 |
| TypeScript DTO type file | 15–30 |
| Repository (5 CRUD methods) | 60–100 |
| Repository (10 methods) | 120–180 |
| Simple service (3–5 methods) | 50–80 |
| Complex service (6–10 methods) | 100–160 |
| API route (1 HTTP method) | 30–60 |
| Server action (1 action) | 20–40 |
| UI component (no state) | 30–60 |
| UI component (with state/hooks) | 60–120 |
| UI component (complex, form) | 100–180 |
| Custom hook | 30–60 |
| Page (simple layout) | 40–80 |
| Page (data fetch + layout) | 80–140 |
| Validator util | 20–50 |
| Unit test file (5–10 tests) | 60–120 |
| Integration test file | 80–150 |

If the sum of estimated LOC across a task's allowed files exceeds 200 → split.

## Splitting Patterns

### Pattern A — Split by HTTP Method

An API route with GET + POST + DELETE → 3 tasks:

```
TASK-NNN   — GET /api/<entities> Route
TASK-NNN+1 — POST /api/<entities> Route
TASK-NNN+2 — DELETE /api/<entities>/[id] Route
```

Each task's Allowed Files: only the route file (modify).
GET and POST are independent. DELETE may depend on GET (if it references types).

### Pattern B — Split by CRUD Method Group

A large repository (10+ methods) → 2 tasks:

```
TASK-NNN   — <Entity> Repository Read Methods   (findById, findAll, findByUserId)
TASK-NNN+1 — <Entity> Repository Write Methods  (create, update, delete)
```

Both touch the same repository file. TASK-NNN+1 depends on TASK-NNN.

### Pattern C — Split by Component Hierarchy

A page that requires multiple new components:

```
TASK-NNN   — <Entity>Card Component
TASK-NNN+1 — <Entity>List Component      (depends on <Entity>Card)
TASK-NNN+2 — /<entity> Page              (depends on <Entity>List)
```

One component per task. The page imports the leaf components.

### Pattern D — Split Schema from Migration

```
TASK-NNN   — <Entity> Data Model         (modifies schema file)
TASK-NNN+1 — <Entity> DB Migration       (runs migration command; creates migration file)
```

TASK-NNN+1 depends on TASK-NNN. The migration task's Allowed Files uses
`[TBD: path generated by TASK-NNN — migration command output]`.

### Pattern E — Split Entity Types from DTO Types

When a type file mixes entity types with request/response shapes:

```
TASK-NNN   — <Entity> Entity Types       (base interfaces)
TASK-NNN+1 — <Entity> DTO Types          (Create<Entity>Input, Update<Entity>Input, etc.)
```

TASK-NNN+1 may import from TASK-NNN output.

### Pattern F — Split Service by Capability

A service that spans multiple unrelated capabilities:

```
TASK-NNN   — <Entity> CRUD Service       (create, read, update, delete)
TASK-NNN+1 — <Entity> Search Service     (fullTextSearch, filterBy<Attribute>)
```

Separate files: `<entity>-service.ts` and `<entity>-search-service.ts`. Independent.

## Merge Conditions (when to combine)

Two concerns MAY be combined into one task only if ALL are true:

1. Total file count ≤ 3.
2. Total LOC estimate ≤ 100.
3. Both concerns live in the same file.
4. Combined task title clearly describes both concerns.

**Valid merge**: A validator and formatter for the same entity in one utility file.

**Invalid merges**:
- Service + Repository (different abstraction layers)
- API route + UI component (different tiers)
- Data model + Types (different lifecycle — schema runs first)

## Story Task Count Guidelines

| Story Size | Expected Tasks |
|---|---|
| Tiny — add a single field or config change | 1–3 |
| Small — new entity, backend only | 3–5 |
| Medium — new feature, full stack | 5–10 |
| Large — multi-entity feature | 10–15 |

If you generate > 15 tasks for one story, the story is too large. Stop and tell
the user to split the story before continuing.

---

## The `--split <TASK-ID>` Command

When invoked with `--split TASK-NNN`:

1. Read the existing `TASK-NNN.md`.
2. Identify which gate(s) it fails.
3. Apply the appropriate pattern above.
4. Write replacement files: `TASK-NNN-a.md`, `TASK-NNN-b.md`, …
5. Update any task that listed `TASK-NNN` in Dependencies to reference the new IDs.
6. Rename the original to `TASK-NNN.md.archived` — NEVER delete it.
